## [Unreleased]

### Added
- **MVP Sub-fase 2 — `harness ui` viewer.** New `src/viewer/` (Bun.serve + vanilla JS SPA, no build step). Boots via `bun run run -- harness ui [--port=N] [--no-open]`, opens browser, lives until Ctrl-C. Read-only HTTP API: `GET /api/health`, `GET /api/config` (exposes `HARNESS_PHOENIX_URL` to the SPA), `GET /api/assistants`, `GET /api/runs` (cross-assistant aggregate), `GET /api/runs/:cwdHash/:slug` (full state + plan + pre-baked Phoenix deep-link URL — Phoenix doesn't expose CORS so the viewer resolves project name → GraphQL ID server-side, cached 30s), `GET /api/runs/:cwdHash/:slug/git-log` (parsed `git log <branch>` from the run's worktree). SPA features: cross-assistant runs list with auto-refresh polling (3s), status/retry-count badges, click into detail showing origin + environment + plan tasks (with summary, status badges, attempts counter, commit_sha, expandable acceptance criteria) + phases timeline (name/attempt/status/duration/session_id) + commits on branch + recent history events + a single "Open trace in Phoenix" button at the header. Tailwind via CDN, hash-based router, ~330 LOC HTML+JS total. Workflows without `plan.json` (e.g. `docs`) skip the Plan section gracefully.
- **MVP Sub-fase 1 — Phoenix observability (opt-in).** New `src/harness/observability/phoenix.ts` integrates Arize OpenInference instrumentation for the Claude Agent SDK. Activated only when `HARNESS_PHOENIX_URL` env var is set; absent → zero-overhead no-op. `setupPhoenix({ workflowId, runId, taskSlug, cwd })` registers `@arizeai/phoenix-otel` once per process and patches the SDK namespace via the documented ESM workaround (shallow-copy the namespace before `manuallyInstrument()` because ESM forbids namespace mutation; validated in `scripts/probes/phoenix/02-openinference.ts`). **Project naming**: `basename(cwd)` (per user preference) — runs from the same repo cluster in one Phoenix project regardless of workflow. **One trace per harness run**: `withRunSpan(setup, taskSlug, attrs, body)` creates a top-level OTel span named after the task slug (so the trace appears as e.g. `calculator` in Phoenix's Traces tab). All phase work runs inside `context.with(runSpan, body)` so phase spans inherit the same `trace_id`. Resource attributes set on every span: `harness.workflow`, `harness.run_id`, `harness.task_slug`, `harness.cwd` (queryable in Phoenix UI). **Phase span renaming**: a custom `RenameClaudeAgentSpanProcessor` registered via `register({ spanProcessors: [defaultProcessor, renameProcessor] })` reads a `PHASE_CONTEXT_KEY` set by `withPhaseContext(setup, phaseName, body)` (called by `sessionRecorder` around each `query()`) and renames the SDK's `ClaudeAgent.query` span to `harness.<phase>` via `span.updateName()`. Result: flat tree `calculator (AGENT) > harness.planner (AGENT) > Read/Bash/Write (TOOL)`, no double-wrapper. **`forceFlush()` on run-span end** (captured `tracerProvider.forceFlush?` at register-time) — without this the BatchSpanProcessor often dropped the run span on process exit. State.json gets `state.phoenix: {project, trace_id}` set after the run span starts; viewer pre-bakes `phoenix.url` into its detail response. Resume runs get a NEW trace (separate process can't continue a closed trace) but stay queryable in Phoenix by `harness.run_id` resource attribute. OTel SDK 2.x note: `addSpanProcessor` was removed; processors must go through `register({ spanProcessors: [...] })` (default OTLP exporter built via exported `getDefaultSpanProcessor`). Probes that drove the design land in `scripts/probes/phoenix/`: `01-sdk-to-phoenix.ts` (raw built-in OTel export — too verbose, 17 spans for one query), `02-openinference.ts` (the chosen path — 5 semantic spans, AGENT/TOOL kinds, structured input/output).

### Added
- **Tier 3b — RunMode foundation (interactive | silent | async).** Three explicit run modes replace the previous TTY-vs-headless bifurcation. New `RunMode = "interactive" | "silent" | "async"` in `src/harness/types.ts`, propagated through `ResolvedHarnessConfig.mode`, `WorkflowContext.mode`, `orchestrator`, and `sessionRecorder`. **Mode resolution precedence**: `--mode` CLI flag > `harness.json` `defaultMode` > workflow `defaultMode` > auto (`process.stdin.isTTY` → `interactive`, else `silent`). Helper `resolveRunMode(cli, file, workflow)` exported from `config.ts`. **Silent mode**: `AskUserQuestion` is stripped from `allowedTools` before `query()` (the agent never sees it); `ctx.askUser` throws `SilentModeError` (workflows decide whether to catch and default, or fail). **Async mode** for the SDK-side `AskUserQuestion` tool: `canUseTool` parks the entire question batch as ONE `pending_questions` row (kind=`ask_user_question_batch`, `options_json` = serialized `AskUserQuestionItem[]`), captures `phase_session_id` + `tool_use_id` + `phase_name` (new schema columns; `PRAGMA user_version = 2`), and returns `{behavior:"deny", interrupt:true}`. The SDK's `for await` loop throws with `subtype=error_during_execution` (verified via `scripts/canusetool-probe.ts`); `sessionRecorder` detects via the captured `parkState`, returns a new `PhaseRunResult.status = "paused_for_user_input"` variant with `parked` payload. `ctx.runPhase` then writes the pending question and throws `PausedForUserInputError`; orchestrator catches and exits cleanly with `status: "waiting_human"`. **Resume**: `harness answer <runId>` accepts three forms: `<text>` (legacy single string), `--json '{...}'` (multi-answer batch validated per-question via `resolveAnswer`), or no arg (interactive walk via `runAskUserQuestionTTY` reused as the multi-question UX). `Workflow.resumeFromAnswer` signature widened to `(ctx, answer: string | Record<string, string>)`; new optional `ctx.resumeMeta = { phaseName, phaseSessionId, toolUseId }` populated only on batch resumes. **`feature-dev` workflow** implements `resumeFromAnswer` for parked planner batches: re-invokes `runPlanner` with `resumeSessionId` and a Q&A prompt prefix, then re-enters `runDevLoop`. **`harness show <runId>`** renders batch parks with a numbered question/option breakdown plus a copy-pasteable `--json` template. Schema migration is additive (`ALTER TABLE ... ADD COLUMN`, NULL for legacy rows). **AbortSignal-driven mid-stream cancellation is intentionally NOT used** — the design treats SDK sessions as disposable and resumes via `resume:sessionId` + injected user message (validated by the probe: the model integrates the prefix without re-asking even when the prior session ended on `error_during_execution`). Throwaway `scripts/canusetool-probe.ts` documents the empirical findings that locked the design.
- **Tier 2 — Run registry + pause/resume** (`f9ba0d3`). SQLite run registry at `~/.harness/runs.db` (`better-sqlite3`, sync API). Schema: `runs`, `run_events`, `pending_questions` — schema version tracked via `PRAGMA user_version = 1`. Every harness run generates a UUID `run_id`, inserts a `runs` row, and emits `phase_start`/`phase_end` events to `run_events` for every phase via the orchestrator's `ctx.runPhase` wrapper. Harness decisions (commit, retry, reset, blocked_fatal) are also captured as run_events via `ctx.audit`. `plan.json` gains a `run_id` field for cross-reference. **`WorkflowContext.askUser({prompt, options?})`** — in TTY mode prompts inline via `readline` and returns `{answered: true, answer}`; in headless mode persists a `pending_questions` row, sets `runs.pending_question_id`, and returns `{answered: false, runId, questionId}` — the orchestrator then sets status=`waiting_human` and exits without committing. **`resumeHarness(runId, answer)`** — loads the parked run, calls `workflow.resumeFromAnswer(ctx, answer)`, updates final status. Three new CLI subcommands: `harness ls [--status X] [--cwd X] [--workflow X]` (table of recent 50 runs), `harness show <runId>` (run detail + events + pending question), `harness answer <runId> "<text>"` (resume a waiting_human run). **`docs` workflow updated** to demo the full cycle: if intent is missing or < 10 chars, calls `askUser` — in headless mode parks and returns `waiting_human`; `resumeFromAnswer` re-enters `runWriteReviewLoop` with the provided answer. **featureDev phases refactored** to use `ctx.runPhase` instead of importing `runPhase` directly from `sessionRecorder.ts` — phase events now appear in the registry for all workflows. **Built via dogfood, 1 reset cycle**: validator caught on attempt 1 that AC3b (phase events for existing workflows) wasn't achievable without migrating featureDev phases off direct sessionRecorder import; attempt 2 fresh-start landed clean.
- **`docs` workflow** (`c224bd6`). Third workflow in the registry: `writer → loop(reviewer)` shape. Writer creates documentation files based on a free-form intent (decides scope and file count itself per architecture guidance in the prompt); reviewer is read-only and approves/rejects with actionable feedback. On reject, the writer's session is resumed (same retry=resume pattern as feature-dev). Commit happens after reviewer pass, harness composes message as `<writer message>\n\ntask=<id>\nreviewer: <evidence>`. `needsBranch:true`, `needsWorktree:true`. allowedTools: writer = `[Read, Edit, Write, Glob, Grep, Bash]`, reviewer = `[Read, Grep, Glob, Bash]` (read-only). Guards: writer=`{noPlanWrites, noGitHistory}`, reviewer=`{readOnly}`. **Built via dogfood — proved the post-cleanup abstraction empirically: only `workflows/docs.ts` (new) + `workflows/index.ts` (registry +2 lines) changed; zero core edits required.** Validator independently exercised the workflow end-to-end: 3 writer↔reviewer cycles in `/tmp/harness-e2e-docs-*` produced two markdown files (CLI-FLAGS.md + ASSISTANT-CONFIG.md) committed on a fresh branch.

### Changed
- **Runtime migrated from npm/tsx → Bun ≥ 1.3.** Dropping the `tsx` layer (Bun runs `.ts` natively) and aligning with Anthropic's post-Oven ownership + Hono's best-supported runtime. `package.json` scripts: `"run": "bun src/runner.ts"` (was `tsx src/runner.ts`). Added `engines.bun: ">=1.3.0"`. Removed deps: `tsx`, `better-sqlite3`, `@types/better-sqlite3`. Added devDep: `bun-types` (tsconfig `types: ["node", "bun-types"]`). Deleted `package-lock.json`; Bun writes `bun.lock`. **Registry ported to `bun:sqlite`** (`src/harness/state/registry.ts`): import flipped to `import { Database } from "bun:sqlite"`; `.pragma("user_version")` reads became `db.prepare("PRAGMA user_version").get()`; `.pragma("user_version = N")` writes became `db.exec("PRAGMA user_version = N")`; all other calls (`exec`, `prepare().run/get/all`) unchanged. Existing `~/.harness/runs.db` (written by better-sqlite3 in Tier 2/3b) is read cleanly — SQLite file format is universal, no data migration needed. Blocker upstream: better-sqlite3 native module not yet supported by Bun (oven-sh/bun#4290 open); bun:sqlite is the documented replacement and is API-compatible for our usage. Known trade-off: the registry is now Bun-locked and won't run on plain Node. Verified: `bun run typecheck` clean, `scripts/guard-probe.ts` 21/21, `scripts/worktree-smoke.ts` 3/3, registry reads prior runs (`PRAGMA user_version=2` preserved), end-to-end harness smoke (`--mode silent` with a tiny task) completed planner+dev+validator cycle and committed on a fresh branch.
- **Compact mode now surfaces per-phase input/output and the initial user prompt.** Previously compact only emitted lifecycle markers (`starting ordinal`, `session=<id>`, phase metadata) — phase events went to the SQLite registry but not stdout, so you couldn't see what prompt was fed into a planner/developer/validator phase nor the validated `structured_output` that came out. `sessionRecorder.runPhaseAttempt` now defines a `logBlock(header, body)` helper (gated on `logMode !== "quiet"`) that emits delimited `[harness:<phase>] <header> >>> / <<<` blocks around each phase's input and output. Input is the raw prompt passed to `query()` (so on resumes you see the feedback delta, which is the right signal). Output covers every exit path: success prints the validated `structured_output` as pretty JSON; the four silent error paths (`no session_id`, `subtype != success`, `no structured_output`, schema validation failure) and the stream-error path each emit `output (error) >>> <msg> <<<`; the async park path emits `output (paused) >>>`. `orchestrator.runHarness` also logs the initial user prompt once at the top via `[harness] user prompt >>> / <<<` (resume path untouched — the prompt is already in the plan). `--quiet` stays fully silent; `--verbose` continues to dump every SDK event via `console.dir` on top of the new blocks.
- **CLI `askUser` polish (Tier 3a).** Numbered options are now selectable by number (`2` resolves to `options[1]`), exact text, or case-insensitive text. The TTY prompt advertises "(number or text)", echoes `\u2192 selected: <option>` after a valid pick, and re-prompts with a clear error on invalid input. `harness answer <runId> <text>` applies the same resolver: if the parked question carries options, the input must match one (errors with exit 1 otherwise) before the resolved option text is forwarded to `resumeFromAnswer`. Free-text questions (no options) pass through unchanged. Shared logic in new `src/harness/askUser.ts` (`resolveAnswer(options, raw)`). No signature changes to `WorkflowContext.askUser`, `AskUserResult`, `pending_questions` schema, or `resumeFromAnswer`.
- **`assistants.json` moved to `~/.harness/assistants.json`** (`02a09d3`, user-global). Paths must be absolute; the runner errors if a relative path is encountered. Decouples user-config from the harness checkout, removes the worktree bootstrap friction (validators no longer need to symlink the file into worktrees), and consolidates user-global state alongside `~/.harness/runs.db`. No backward-compat fallback. `assistants.example.json` updated.
- **DEVELOPER_PROMPT discipline rule added** (`02a09d3`). Devs must update the relevant CLAUDE.md in the same commit when they modify a public API surface (WorkflowContext type, Workflow type, exports from `workflow.ts`, capabilities in `state/`, harness CLI flags, key file paths). Stops ambient context drift between commits.
- **VALIDATOR_PROMPT efficiency rules sharpened** (`02a09d3`). Three new heuristics: (1) plan nested invocations explicitly before running them and consolidate ACs into the fewest unique runs; (2) use distinct task slugs across nested smokes (or `harness clean` between them) to avoid branch collisions; (3) explicit exception to independence — artifact-based evidence is acceptable for behavior-across-N-modes ACs IF validator confirms artifacts match current diff, runs at least one mode end-to-end, and explains the reasoning.
- **`src/harness/` reorganized** (`d61d59e`). 14 root files → 7 + 2 folders. `plan.ts`, `audit.ts`, `problem.ts` grouped into `src/harness/state/` (everything that writes under `.harness/<slug>/`). `jsonSchema.ts` (10 lines, single caller) inlined into `sessionRecorder.ts` as a private function; `defaults.ts` (16 lines, single caller) inlined into `config.ts` as `GENERIC_DEFAULTS`. CLAUDE.md key paths section refreshed. Validated: typecheck clean, 21/21 guard-probe + 3/3 worktree-smoke pass.
- **Tier 1b boundary cleanup — A1 + A2 + A3.** Post-refactor review surfaced 3 categories of leaks between core and workflows; this pass closes all of them. **A1 (open closed unions):** `PhaseName`, `PlanTaskHistoryEntry`, and `AuditEntry` are now open shapes (workflows declare their own typed history entries / audit entries locally and cast on read; core only requires the discriminator + common fields). **A2 (move workflow-specific symbols out):** `PlannerVerdictSchema`/`DeveloperVerdictSchema`/`ValidatorVerdictSchema` + types moved from `verdict.ts` → `workflows/featureDev/verdicts.ts`; `applyPlannerVerdict` + history formatters moved from `plan.ts`/`phases/developer.ts` → `workflows/featureDev/plan.ts`; `PLANNER_PROMPT`/`DEVELOPER_PROMPT`/`VALIDATOR_PROMPT` + `DEFAULT_PLANNER`/`DEVELOPER`/`VALIDATOR` moved from `defaults.ts` → `workflows/featureDev/defaults.ts`; `phases/{planner,developer,validator}.ts` moved to `workflows/featureDev/phases/`; `TriageVerdictSchema` + `createTriagePlanTask` + `applyTriageVerdict` + new `DEFAULT_TRIAGE` inlined into `workflows/issueTriage.ts`. `verdict.ts` renamed to `jsonSchema.ts` (only `toJsonSchema` remains). **A3 (capabilities coupling):** `Workflow` gains `phaseDefaults: Record<string, ResolvedPhaseConfig>`; `HarnessConfigFile` and `ResolvedHarnessConfig` flatten to `phases: Record<string, PhaseConfig|ResolvedPhaseConfig>` (replaces `planner`/`developer`/`validator` fields). `loadHarnessConfig(cwd, workflow)` is now workflow-aware: deep-merges `workflow.phaseDefaults` with `harness.json`'s `phases` map. `ctx.runPhase` resolves `config.phases[phase]` automatically — no more `config.developer` fallback (errors descriptively if phase config is missing). `harness.example.json` restructured under `phases` key (no backward-compat shim — no real consumer yet). `Plan.planner_session_id` removed in favor of generic `Plan.metadata: Record<string, unknown>`; feature-dev writes `metadata.planner_session_id`. **Guard hooks parameterized:** `buildGuardHooks` now takes `guards: PhaseGuards` (`{readOnly?, noPlanWrites?, noGitHistory?}`) instead of phase-name literals. Each phase declares its own guards: feature-dev planner=`{readOnly}`, developer=`{noPlanWrites, noGitHistory}`, validator=`{readOnly}`; issue-triage triage=`{noGitHistory}`. Net result: a future bug-fix or docs workflow can now be added by writing a single `workflows/<name>/` folder with zero edits to core files. Validated: typecheck clean, 21/21 guard-probe cases pass, 3/3 worktree-smoke tests pass, end-to-end feature-dev run committed `chore: add post-cleanup.txt marker file` on a fresh branch and confirmed `metadata.planner_session_id` is written.

### Added
- **Phase 2 Tier 1b — Workflow abstraction** (`a5e2445`, `e552e82`). Generalizes the harness from a hardcoded `planner→loop(dev,validator)` orchestration into an interpreter that runs any workflow declared via `defineWorkflow({...})`. New `Workflow<TInput>` type: `{id, needsBranch, needsWorktree, inputSchema?, run(ctx)}`. `WorkflowContext` exposes capabilities to the workflow body: `updatePlan(mutator)` (atomic), `audit(entry)`, `currentSha/commit/resetHard/cleanUntracked`, and a generic `runPhase<T>()`. `orchestrator.ts` shrunk 484→152 lines and contains zero workflow-specific logic — just resolves workflow from registry, conditionally creates branch/worktree per flags, builds context, calls `workflow.run(ctx)`, cleans up. Two workflows shipped: `feature-dev` (the existing flow as a ~22-line manifest with private helpers below) and `issue-triage` (new template, `needsBranch:false`, `needsWorktree:false`, allowedTools `[Bash, Read, WebFetch, Grep, Glob]` — decides via `gh issue view <url>` and persists `{action, target_url, payload, reasoning}` into `plan.tasks[0].output`; does NOT execute the action). CLI: `--workflow <name>` selects from registry, `--harness` becomes alias for `--workflow feature-dev`, `--input <path>` reads JSON validated against `inputSchema`. Full self-build via dogfood: harness produced both commits in ~60min, both tasks PASS attempt 1 with independent empirical validator runs (feature-dev wrote `hello.txt` + commit; issue-triage fetched real GH issue + landed decision).
- **Phase 2 Tier 1a — Worktree isolation** (`765afa7`). `IsolationMode = "worktree" | "inline"` (default `worktree`). Each task runs in a dedicated git worktree at `<primary>/.harness/worktrees/<slug>/`; `.harness/<slug>/` state stays in the primary so audit survives worktree removal. Auto-remove on done; preserve on fail/blocked/exhausted for debug. CLI flag `--isolation`, harness.json field `isolation`. `assertCleanTree` is gated on `isolation === "inline"` — primary's cleanliness is irrelevant in worktree mode. Path-anchor refactor: `primaryCwd` (state writes) vs `phaseCwd` (SDK + git ops) propagated through `runPhase`, all phase wrappers, plan/audit/problem helpers, guard hooks. New git primitives: `addWorktree`, `removeWorktree`, `assertWorktreePathAbsent`. Smoke test `scripts/worktree-smoke.ts` covers primitives, sequential runs (2nd doesn't trip on 1st's leftover state), and concurrent runs (Promise.allSettled, two worktrees from same primary). Built off-harness after 5 dogfood attempts each exposed a fresh infrastructure bug.
- **Phase 2 Tier 5 (partial) — Three-mode logger** (`ce9c0f5`, merged in `daa8c4c`). `LogMode = "compact" | "verbose" | "quiet"`. `--compact` (new default) surfaces meaningful orchestrator progress only (phase transitions, validator verdict one-liner, decision, commit SHA + subject). `--verbose`/`-v` keeps the full SDK event dump. `--quiet` prints only final status + branch. Threaded through orchestrator, phases, sessionRecorder. TodoWrite agent capture still pending.
- **`harness clean <slug>` CLI subcommand** (`85474cd`). Idempotent cleanup: removes worktree (force), deletes branch, removes `.harness/<slug>/`. Each step a no-op on absent state. Recovers from killed/crashed runs without manual git plumbing.
- **`.harness/.gitignore` tracked statically** (`*` + `!.gitignore`) — replaces the prior runtime-write pattern that race-conditioned with `assertCleanTree` on subsequent runs. Backed in `74ca765`.
- **Transient API error retry** in `runPhase` — up to 3 attempts with exponential backoff (2s/4s/8s, cap 30s) for `overloaded_error`, `rate_limit_error`, HTTP 5xx. Each retry starts a fresh attempt record (preserves prior session file as evidence).
- **Validator escape hatch** in guard hooks — `Write|Edit|MultiEdit|NotebookEdit` to paths outside `phaseCwd` are now allowed (e.g. `/tmp/harness-e2e-*`) so validators can set up empirical fixtures. Primary repo itself stays strict read-only.
- **Developer git-committer escape hatch** — `cd /tmp/...`, `cd /private/tmp/...`, `cd /var/folders/...`, or `git -C <path>` where path resolves outside `phaseCwd` are allowed (throwaway test repos). Forbidden git regex broadened to catch `git -C <path> commit` patterns. Guard probe extended to 21 cases (escape-hatch + sneak-attempt coverage).
- **`PHASE2.md`** — roadmap with status tracker, gating questions resolved/open, evolutions during implementation, and backlog discovered through real dogfood usage.
- **Phase 2 Tier 0**: three invariant guard hooks registered per-phase in `sessionRecorder`. Validator `PreToolUse` denies `Write|Edit|MultiEdit|NotebookEdit` (read-only invariant). Developer `PreToolUse` denies writes to `.harness/<slug>/plan.json` (sole-writer invariant) and Bash commands matching history-changing git operations (`commit|push|reset|rebase|merge|revert|cherry-pick|tag|am` or `--amend`) (sole-committer invariant). Unit-tested via `scripts/guard-probe.ts` (13/13 cases).
- **Phase 2 Tier 0**: problem annotation capture. `DeveloperVerdictSchema` and `ValidatorVerdictSchema` gain an optional `problems: Problem[]` field (`{category, severity, description}` with categories `environment | design | understanding | tooling`). Orchestrator writes one file per problem to `.harness/<slug>/problems/<id>.json` — idempotent, append-only, safe under git merges. Schema versioned at `schema_version: 1`. Dataset accumulates from day one for the future `/improve` consumer.
- `PHASE2.md` — roadmap with 11 tiers, gating questions, completion criteria. Drives session-by-session pull-down of Phase 2 work.
- Empirical probe (`scripts/hook-probe.ts`) proving hooks DO fire in Single Message Input mode, contradicting a claim in the streaming-vs-single doc. Streaming mode is therefore not a prerequisite for Phase 2.
- Validator now reports a third signal `recommend_reset` alongside `verdict` and `reasons`. When true, the harness wipes the working tree and starts the next attempt fresh instead of resuming.
- `maxRetriesBeforeReset` config (default 1) — after N failed retries the harness forces a reset regardless of the validator's recommendation.
- `audit.jsonl` append-only log at `.harness/<slug>/audit.jsonl` with every developer outcome, validator verdict, harness decision (retry/reset/commit/failed/blocked_fatal), and commit/reset execution records. Task-local `.gitignore` excludes it.
- Retry path resumes the developer's previous session (`resume: sessionId`) with only the new validator feedback in the prompt; reset path starts a fresh session.
- `commit_sha` at the task level in `plan.json` — records the single commit that represents the accepted task.

### Changed
- **Default model swapped to `sonnet`** for all three phases (`fdbf602`). Opus 4× more expensive and noticeably slower; Sonnet proven sufficient for self-build dogfood. Per-project `harness.json` can override back to `opus` for production use.
- **Default `permissionMode` swapped to `bypassPermissions`** (`32f76dd`). The previous `auto` silently fell back to "ask user" in headless / background runs, which never gets answered — Write/Edit calls to `/tmp/harness-e2e-*` paths sat blocked. Guard hooks already enforce the invariants we care about; stacking the SDK's own permission layer on top added only silent failures.
- **Default maxTurns bumped** to planner=50 / developer=200 / validator=200. The previous 10/30/20 were empirically insufficient for non-trivial work — planner ran out exploring a medium codebase, validator ran out doing nested empirical runs under the independence contract. Real-world agentic engineering tasks span many turns; these limits are safety caps, not targets.
- **VALIDATOR_PROMPT tightened twice during dogfood**: (1) explicit "INDEPENDENT empirical exercise required — developer artifacts are one input, never a substitute for your own run", (2) "ONE comprehensive nested run per task max; for cosmetic/additive changes, structural review + smoke run is sufficient — don't recurse the harness per AC". The second pass was added after observing 4× full nested invocations from a single validator burning 9+ minutes on one Bash call.
- **PLANNER_PROMPT tightened**: explicit task-granularity guidance — default to the smallest viable plan; 1 task is right for narrow refactors and additive features; 2-3 only for distinct validation surfaces; 4+ ONLY for genuinely independent shippable units. Cost framing stated in-prompt so the planner has the right objective function. Added after observing the planner default-splitting "add a logger flag" into 4 tasks, multiplying validator cost.
- **DEVELOPER_PROMPT augmented** with idempotency-check rule: when the task changes how the harness is invoked (new flags, modes, preconditions, isolation, state on disk), smoke tests MUST run the harness twice consecutively — many bugs only surface on the second run when the first leaves untracked state behind.
- **Commits now happen only after validator pass.** The developer no longer commits during its phase; it proposes a conventional-commit `commit_message` in its structured output. The harness composes the final commit as `<developer message>\n\ntask=<id>\nvalidator: <evidence>` and creates it from the working tree.
- Reset path uses `git reset --hard <pre-phase-sha>` + `git clean -fd` (ignored files survive, so `.harness/<slug>/` sessions + audit log are preserved).
- `blocked` from the developer is now **fatal** — the whole plan is marked `failed` and the loop aborts. Rationale: if the agent could have unblocked itself it would have; if it couldn't, human triage is required. This is temporary until human-in-the-loop feedback exists.
- On task fail (exhausted retries) or blocked, the working tree is reset to the pre-phase SHA so the branch only shows committed work.
- Developer prompt no longer instructs the agent to commit; validator prompt notes that changes are uncommitted when it runs.

### Fixed
- `.gitignore` pattern `node_modules/` (with trailing slash) only matched directories — symlinks named `node_modules` slipped through and got tracked when an agent created one inside its worktree to make `npm` work there. Tightened to `node_modules` (no trailing slash) catches files, dirs, and symlinks. The accidental tracked symlink was also dropped (`74ca765`).
- Planner `maxTurns: 10` was too low for non-trivial codebases — bumped twice during dogfood, ultimately settling at 50.

## [0.1.0] — 2026-04-20

### Added
- CLI launcher (`src/runner.ts`) built on `@anthropic-ai/claude-agent-sdk`. Streams every SDK event into a per-session JSON file.
- `assistants.json` registry of named working directories, selected via `--assistant <name>`. Paths resolve relative to the file itself. A gitignored config with `assistants.example.json` documenting the schema.
- `--verbose` / `-v` flag that dumps every SDK event as it arrives.
- Three-phase harness orchestration (planner → loop(developer → validator)), activated by `--harness`. Planner runs once; dev/validator loop until the plan is complete or iteration caps are hit.
- `--task <slug>` with a timestamp fallback. The harness creates a `harness/<slug>` branch, commits `plan.json` after planning, and commits once per accepted task.
- `plan.json` lives at `<cwd>/.harness/<slug>/plan.json` and is written only by the TypeScript layer. Agents return Zod-validated structured verdicts that the harness merges in.
- Precondition checks before any work (is git repo, clean tree, branch absent). Fail fast on any violation.
- Per-project `harness.json` at the target repo root overrides defaults per phase (prompts, tools, model, effort, `maxTurns`, MCP servers). Deep-merge, arrays replace. Library defaults live in `src/harness/defaults.ts`.
- Structured outputs via the SDK's native `outputFormat: { type: "json_schema", ... }`. Schemas defined as Zod in `src/harness/verdict.ts`; `z.toJSONSchema()` feeds the SDK.
- Ordinal prefix on session files (`NNNN_<session_id>.json`) so execution order is obvious without parsing UUIDs.
- `.gitignore` inside each task directory so `sessions/` stays untracked while `plan.json` is versioned.
- `harness.example.json` documenting the per-project config schema.

### Fixed
- `toJsonSchema()` strips the top-level `$schema` key produced by `z.toJSONSchema()`. The bundled `claude-code` binary silently ignores any schema that has `$schema` at the top, causing `structured_output` to come back undefined with no error — manifested as the planner "succeeding" with free-form text instead of the validated plan.
